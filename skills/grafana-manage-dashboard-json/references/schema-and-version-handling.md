# Schema and Version Handling

Use this file when the task is mostly about identity, schema, or revision fields rather than panel content.

| Field | Meaning | Keep or edit? |
| --- | --- | --- |
| `uid` | Durable dashboard identity | Preserve when updating the same dashboard |
| `id` | Database-generated identifier | Treat as generated, not portable identity |
| `schemaVersion` | Dashboard schema revision | Preserve unless the source workflow explicitly changes it |
| `version` | Dashboard revision in classic workflows | Preserve for updates, but do not confuse it with schema revision |
| `metadata.resourceVersion` | Concurrency token for the new API resource | Include only when the new API update workflow requires it |

## Important Distinctions

- `uid` is the durable identity. It is the field to preserve when the intent is "this is still the same dashboard."
- `schemaVersion` is a schema revision field. It is not the same thing as dashboard `version`.
- `schemaVersion` is not a reliable signal for panel modernity. A higher schema version can still contain legacy panel types such as `graph` or `singlestat`.
- `version` is the dashboard revision used by classic dashboard update flows. It is not the same thing as `resourceVersion`.
- `id` is generated by Grafana's database layer and should not be treated as a reusable identity field.
- `resourceVersion` belongs to the new API resource model, not to classic dashboard JSON.

## Common Mistakes

- Confusing classic numeric `schemaVersion` with schema v2 as a dashboard format choice
- Reusing classic `version` rules when building a new API resource body
- Preserving classic `id` in reusable files or templates
- Dropping `uid` accidentally when the user intended to update an existing dashboard
