# Safe Editing

Start from a known-good export or payload before making structural edits. Change the smallest possible JSON surface that solves the task.

## Safe to Edit Deliberately

- Dashboard content such as `title`, description, tags, links, annotations, variables, queries, and layout fields that match the identified dashboard shape
- Classic `panels[].targets`, `panels[].gridPos`, and related presentation fields when the task is explicitly about them
- Schema v2 `elements`, `layout`, `timeSettings`, and `variables` when the artifact is actually schema v2

## Treat as Managed or Generated

- Classic `id`
- Classic `version` when cloning or templating, unless the task is specifically about updating an existing dashboard revision
- New API metadata fields such as `metadata.resourceVersion` and other server-managed metadata
- Wrapper fields that exist only for a specific transport path, such as legacy API `overwrite` or folder fields

## Durable Identity Rules

- Preserve `uid` when updating the same dashboard across exports, provisioning, or API payloads.
- Change or remove `uid` only when the user is intentionally creating a different dashboard identity.
- Do not use classic `id` as identity in portable JSON.

## Editing Discipline

- Do not rename or remove fields whose meaning is unclear.
- Do not mix layout changes, query changes, and transport-wrapper changes unless the task needs all of them.
- If a field appears generated by Grafana, explain whether you are preserving it, dropping it, or leaving it untouched.
- When uncertain, describe the risky field instead of inventing semantics for it.
